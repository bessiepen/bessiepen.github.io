post和get/method

发表于 2016-03-28   |   分类于 技术笔记   |  

什么是post、get

GET/POST是在网页中提交form数据时的HTTP方法。

HTTP是超文本传输协议，用于客户端和服务器之间的通信，方式是客户端向服务器发出请求，服务器收到请求后向客户端返回响应。比如，在百度网页中写入关键字，点击搜索，那么浏览器这个客户端就向百度的服务器发送了一个包含关键字的网页搜索请求，服务器收到后把包含关键字的网页信息返回给客户端，页面上就显示了搜索后的结果。

GET是从指定的资源上请求数据，如搜索。
POST是向指定的资源提交数据，如上传图片。

它们的区别有哪些呢？

字符串（名称/值对）的发送方式不同：

GET方式：将字符串（name/value）生成到URL中发送，以“？”连接URL和字符串，并列的字符串用”&”连接，字符串的name和value用”=”连接，URL是form元素中action指向的值。格式为：

test/demo_form.php?name1=value1&name2=value2

POST方式：将字符串（name/value）生成到URL的head主体中包装发送，不会显示在URL中，格式为：

POST test/demo_form.php HTTP/1.1
Host: 发送数据的网页
name1=value1&name2=value2

安全性不同：

由于第一点的发送方式不同，GET的数据直接显示在URL中，是可见的，易泄露，所以敏感数据一定不能使用GET方式。
POST方式下将数据封装在URL的头部中，不可见，更安全。

GET请求有缓存，能被存储到浏览器浏览历史中，能被保存为书签。
POST请求无缓存，不能被存储到浏览器浏览历史中，不能被保存为书签，数据不会保存在web服务器日志中。

字符及编码不同：

GET方法只能对ASCII字符集进行编码。编码类型为：application/x-www-form-urlencode
POST方法对字符类型无限制。编码类型为：application/x-www-form-urlencode 或者multipart/form-data。

注意: form中的enctype属性为multipart/form-data时，method只能使用POST值。

数据长度要求不同：

GET方法将数据生成到URL中，整个URL的长度不能超过2048个字符。
POST对数据长度无限制。所以，上传图片这样数据大的只能用POST方法。

Outline:
html文档根据header的层级系统来向用户显示文档结构大纲，形成一个目录使用户能快速概览整体结构。大纲在html中的位置置于<body> </body>之间。

Outline Algorithm:

大纲算法允许用户代理(user agent)从一个web页面生成一个信息结构目录，让用户对页面有一个快速的概览。类似书籍、PDF、帮助文档等，都有一个清晰的目录结构，用户能方便的定位所需内容。一个良好结构的大纲，不仅是对于搜索引擎的优化，更是为借助于屏幕阅读器浏览网页的盲人（或弱视力）用户提供了巨大的帮助。
——七宝《HTML大纲算法对结构的影响》

CSS:font-size value: px em %

发表于 2016-03-26   |   分类于 技术笔记   |  

Value

px

px是固定值，不过针对于显示器的小点来说，也可以算是相对值。
浏览器默认font-size为16px。

em

em是相对值，它是倍数关系。当浏览器的默认font-size=16px时，16px=1em。当浏览器的默认font-size=20px时，20px=1em。

%

%是相对值，相对于父元素的font-size值。比如，父元素的font-size=16px，子元素设置了font-size=200%，则子元素的字号为16px * 200%=32px。

px和em的换算

主要是把px换算成em，这样做是因为em是相对值，能够在不同分辨率的情况下自适应字体大小。
换算公式为：

px/16=em

px指想要的px具体值，如16px，24px，32px等。
16指当前的计算标准，即父元素的font-size值，不一定就是16px，而是看最近的父元素的字号是多少。
那么，就可以换算出当想要的px值是16px，父元素的字号也是16px时，等价的em值为：

16px/16=1em

同理：

想要的px值 父元素字号 等价的em值
14px 16 14px/16=0.875em
10px 16 10px/16=0.625em
14px 14 14px/14=1em
10px 14 10px/14=0.714em

反过来，知道了em值，知道了父元素的字号，就可以倒算回等价的px值，如：

1em*16=16px

px、em、%三者之间的换算应用

以给<body>设置css：body{font-size:62.5%}为例。

之前遇到了字体无法自适应的问题，网上搜了解决方法，就是将<body>字号设为62.5%，然后此时的1em正好等于10px，便于px与em之间的换算。那么62.5%是怎么来的呢？

看之前上面那个表格的第三行就知道了，10px/16=0.625em。但是这个推理过程慢慢来看：

首先，我们的目的是要让换算标准为10，因为当换算标准为10的时候，意味着14px=14px/10=1.4em，26px=26px/10=2.6em，这样把px换算成em就非常方便，心算就能出来了。
其次，怎样使换算标准为10？上面说到，换算标准就是父元素的字号，我们要应用到整个页面字体，此时最大的父元素就是<body>，也就是<body>的字号应该是换算标准，所以它应该是10px。我们又知道，浏览器的默认字号为16px，所以浏览器的字号是继承给<body>的，我们又知道，元素字号的百分比值是根据父元素的字号来乘以该百分比的，即<body>的字号应该是：浏览器字号16px*百分比=10px，这样，这个百分比=10px/16px=62.5%。即body{font-size:62.5%}。

根据这个结论，我们来顺着验证下：

序号 子元素 父元素 子元素的字号值换算
第一步 body 浏览器 百分比与px的换算
字号值 62.5% 16px body：62.5%*16px=10px
第二步 div body px与em的换算
字号值 16px 10px（由上一步得到） div：16px/10=1.6em

这样就最终达成了所有子元素与父元素字号px和em之间的换算标准为10。这样我们再来重新理解一下这个css设置body:{font-size:62.5%}，它其实是把整个内容字体缩小到62.5%，再根据em的倍数来进行放大到合适的px值。比如，本来是16px，现在缩小62.5%变成了10px，但是我们还是想要16px，于是又把10px放大1.6倍（em）变成了16px。为什么要经过62.5%这道中间程序，完全是为了得到10这个换算标准，便于计算而已。

然而，它有一个大问题：这个设置能使所有的计算标准都为10px吗？

问题出来了，当更深层嵌套的元素使用em时，它的计算标准是根据自己的上一级父元素的字号，而不是基于<body>的10px。而上一级父元素的字号已经不是10px了，可能是任何数值，那么此时要设置自适应的em，依旧需要公式：px/标准=em。
这样一来，为body设置font-size:62.5%以达到10px的计算标准来便于计算的目的并不是完整的解决了所有的计算。它的实用性被削弱了，还对不清楚的人造成了计算深层嵌套元素字号的混乱和麻烦。
因此，想设置自适应，不如老老实实的遵循最基本的原理，也就是这个公式：

px/父元素字号=em

有些属性如line-height,text-indent，当它们本身的值的单位为em时，就需要根据元素的font-size值为基础计算。

###当元素的font-size为固定值px时
如：

div的line-height=16px2=32px；
div的text-indent=16px2=32px。

当元素的font-size为相对值em时

如：

首先，计算div的font-size=父元素字号设16px2=32px；
然后，计算div的的line-height=32px2=64px；
同理，计算div的text-indent=32px*2=64px。

line-height的value换算问题

value为em或%时

元素的line-height的px值以本元素的font-size为标准计算，上一小节已经说到了。
此时，如果其子元素不单独设置line-height，该元素的line-height的值将继承给子元素，且 与子元素的font-size无关。

此时，<p>的line-height=10px*2em（或200%）=20px。

value为系数（一个无单位的数值）时

元素的line-height的px值以本元素的font-size为计算标准。
但是，如果其子元素不单独设置line-height，该元素的line-height的系数将继承给子元素，且 与子元素的font-size有关。

此时，<p>的line-height=12px*2=24px。

CSS:包含块( containing block)

发表于 2016-03-04   |   分类于 技术笔记   |  

包含块是一个元素进行定位和尺寸计算的参照框，它是一个矩形。例如对元素p定位left:20px，那么这个left:20px是针对什么的左边20像素？那就是包含块的边界位置。

包含块的判定：

无position属性的元素（也就不涉及偏移量）：

元素的包含块是该元素上一级父元素块级框的内容边界（content edge），例子：

<html><body><div><p></p></div></body></html>

元素 包含块
html 初始包含块
body html
div body
p div

有position属性的元素：

  • static：不定位，left等无效，无包含块；
  • relative：包含块为该元素原本在的内容边界；
    • 例子：<p id=“relative”></p>，p元素的包含块是p元素初始内容框
  • fixed：
    • 设置偏移量时，包含块为当前视窗本身；
    • 未设置偏移量时：
      • 无已定位的祖先元素，会定位到最近父元素的左上角；
      • 有已定位的祖先元素
        • 祖先元素无偏移量，会定位到祖先元素的左上角。
        • 祖先元素有偏移量
          • 祖先元素为position：relative
            • 祖先元素是父元素时，偏移量不会继承下来，该元素还是定位到父元素的左上角固定不动；
            • 祖先元素是父元素以外的祖先元素时，偏移量会继承下来，但该元素以视窗为偏移。
          • 祖先元素为position：absolute/fixed
            • 不论父元素还是祖先元素有偏移量，会继承偏移量，但该元素以视窗为基准偏移。
  • position：
    • 设置偏移量时
      • 该元素附近有position值为static或无已定位的祖先元素，其包含块为初始包含块 <html>。
      • 该元素附近有position值为非static的祖先元素，包含块为该祖先元素：
        • 祖先元素是块级元素：包含块为祖先元素的padding edge形成；
        • 祖先元素是内联元素：包含块为祖先元素的内容边界形成。（direction:ltr/rtl，不同时判定不同，这点我还没搞清楚，以后遇到再补）。
    • 未设置偏移量时
      • 无已定位的祖先元素，本元素会定位到最近父元素的左上角；
      • 有已定位的祖先元素：
        • 祖先元素未设置偏移量，本元素会定位到祖先元素的左上角
        • 祖先元素已设置偏移量，本元素会继承偏移量，以祖先元素为偏移基准。
